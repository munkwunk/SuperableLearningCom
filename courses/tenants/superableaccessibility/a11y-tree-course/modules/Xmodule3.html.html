Your Toolkit for Building a Better Tree

Understanding the two blueprints is one thing, but building them correctly is another. The good news is that you don't need complex tools to get started. The best practices are simple, powerful, and built right into the web.

In this module, we'll cover the golden rule of accessible development and show you how to use your browser's built-in tools to see your work through the "eyes" of assistive technology.

The Golden Rule: Semantic HTML First

If you remember only one thing from this course, make it this: Use the right HTML element for the right job. This is called using semantic HTML.

When you use an element like <button>, <nav>, or <h1>, you're using code that has meaning baked right in. The browser already knows what these elements are and how they should behave.

Think about a <button> element. When you use it, you get a huge amount of accessibility for free:

  • Correct Role: The browser automatically adds the "button" role to the Accessibility Tree.
  • Keyboard Focus: It's naturally focusable with the Tab key.
  • Keyboard Operation: It can be activated with both Enter and the Spacebar.

This principle applies even if you're not writing code by hand. If you're using an authoring tool like Storyline or a website builder, this means choosing the tool's built-in "Button" feature instead of drawing a generic shape and adding a click action to it.

When you use a non-semantic <div> with a click event instead, you get none of that built-in functionality. Visually, it might look the same, but to the Accessibility Tree, it's just a generic container with no meaning. You would have to manually add the role, the keyboard focus, and the event listeners—a process that is complex and easy to get wrong.

When to Use ARIA (The "Spackle")

Sometimes, you'll build complex components like custom menus or tab panels that don't have a perfect native HTML equivalent. In these cases, you can use WAI-ARIA (Accessible Rich Internet Applications) to add the missing semantics.

But ARIA should be your last resort, not your first choice. A great way to think of it is that HTML is the rebar, and ARIA is the spackle. You use rebar to build the strong foundation of a house; you use spackle to patch small holes.

Before you reach for ARIA, use this simple decision tree:

ARIA Decision Tree (Text Alternative)

A flowchart for deciding when to use ARIA.

  1. Start with the question: I need to build a UI component.
  2. Next, ask: Is there a native HTML element that does this?
    • If the answer is Yes: Use the native HTML element. For example, use <button>, <input>, or <details>.
    • If the answer is No: Then, you should consider using ARIA to add the missing semantics. For example, add role="tab".
