Fake Links (Anti-Pattern)

Links are the basic building blocks of the web. They are used for navigation between pages and sections. Using a <span> or <div> to simulate a link is a major accessibility anti-pattern.

Practice Task

1. Open the screen reader's elements list (e.g., NVDA Elements List with NVDA + F7).

2. Identify which elements appear as links and which functional cues distinguish them.

3. Navigate to the native link and verify the functional cues provided when it is focused or activated.

Why it's broken

  • Not Focusable: Keyboard users skip right past the fake link.
  • No Role: Screen readers don't announce "link," so the user doesn't know it's a navigational element.
  • No URL context: Real links show the target URL in the status bar, providing confidence to the user. Fake links provide no such feedback.

Compliance Failures

  • SC 4.1.2: Name, Role, Value - The span lacks a link role, so its purpose is not announced to screen readers.
  • SC 2.1.1: Keyboard - The span is not focusable and cannot be activated via keyboard.

Accessible Comparison

Like buttons, links are best handled by native HTML. Notice how the native <a> tag gives you everything for free.

Key Differences

  • Span: A total barrier for navigation. Keyboard users cannot reach it, and it has no semantic meaning.
  • ARIA Polyfill: Requires role="link" and tabindex="0". This is reachable via Tab and Enter, but fails to appear in standard "Links List" dialogs in some browsers.
  • Native Link: Automatically focusable, has the "link" role, and is part of the "Links List" in screen readers (NVDA + F7). This is the gold standard.

Compliance Success

  • SC 4.1.2: Name, Role, Value - Using a native <a> tag ensures the correct role and name are announced.
  • SC 2.1.1: Keyboard - Native links and properly polyfilled controls are focusable and respond to the Enter key.
