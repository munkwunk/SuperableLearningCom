Form Validation & Errors

Forms are the primary way users provide data. When that data is incorrect, the application must provide clear, programmatically associated feedback to help the user fix the issue. (Note: This interaction is designed to demonstrate validation failures; there is no way to submit a "successful" form in this demo.)

Practice Task

1. Attempt to submit the form with empty fields.

2. Identify the functional cues that indicate an error state and how focus is managed to guide the user.

3. Use the error summary links and identify the cues that confirm you have moved to the related input.

Inaccessible Example

In this broken version, errors are indicated only by a vague message and color changes. Focus remains on the button, and the specific invalid field has no text-based error identification. A screen reader user is left entirely unable to identify what went wrong.

Please fix the errors below.



Why it's broken

  • Color Alone: The Username field is indicated as invalid only by its red border. Screen readers do not announce CSS border colors, so this state is invisible to NVDA users.
  • Vague Messaging: The message "Please fix the errors below" doesn't specify which field is broken or how to fix it.
  • No Initial Focus: When the button is pressed, the screen reader remains silent and focus stays on the button. The user has no idea that an error message appeared at the top of the form.
  • No Programmatic Link: There are no aria-invalid or aria-describedby attributes to link the error state to the inputs.

Compliance Failures

  • SC 1.4.1: Use of Color - Color is the only visual cue provided to identify the specific field that requires attention.
  • SC 3.3.1: Error Identification - The specific error is not identified in text, and the vague summary provides no actionable information for screen reader users.
  • SC 4.1.3: Status Messages - The appearance of the error message and the change in form state are not announced to the user.

Accessible Example

The accessible version uses aria-invalid, aria-describedby, and automatically focuses the first error on submission.

How it's fixed

  • Aria-invalid: Tells the screen reader the field needs attention.
  • Aria-describedby: Directly links the error message ID to the input, so NVDA reads the error as soon as the field is focused.
  • Error Summary: Provides a list of all errors at the top of the form, with links to jump directly to the invalid fields.

Compliance Success

  • SC 3.3.1: Error Identification - aria-invalid ensures the error state is programmatically identifiable.
  • SC 3.3.3: Error Suggestion - aria-describedby links corrective suggestions directly to the input.
  • SC 4.1.3: Status Messages - An error summary and focus management provide clear status updates upon submission.
