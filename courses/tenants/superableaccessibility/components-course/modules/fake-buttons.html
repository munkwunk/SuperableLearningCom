Fake Buttons (Anti-Pattern)

The "First Rule of ARIA" states: "If you can use a native HTML element or attribute with the semantics and behavior you require already built in, then do so." Using a div as a button is one of the most common and damaging accessibility failures.

Practice Task

1. Try to activate the "Fake Button (Div)" using only your keyboard and note the lack of focus or interaction cues.

2. Tab to the "Native Semantic Button" and identify the functional cues that indicate its role and state.

3. Tab to the "ARIA Polyfilled" version and verify it provides the same functional cues as the native button.

Inaccessible Example

In this broken version, we use a div. It looks like a button, but it is invisible to keyboard navigation and screen reader roles.

Fake Button (Div)

Why it's broken

  • No Tab Order: A div is not a tab stop. Keyboard-only users cannot reach this control.
  • No Keyboard Activation: Native buttons respond to Enter and Space. A div with a click handler does not.
  • No Role: NVDA announces this as "text," not a "button," so the user doesn't know it's interactive.

Compliance Failures

  • SC 4.1.2: Name, Role, Value - The div lacks a button role, so its purpose is not announced to screen readers.
  • SC 2.1.1: Keyboard - The div is not focusable and does not respond to keyboard activation (Enter/Space).

Accessible Comparison

Compare the three approaches below. Notice how the native <button> is the simplest and most robust solution.

Key Differences

  • Div: Completely unusable for keyboard-only users. Interestingly, an NVDA user can technically activate it using arrow keys (Browse Mode), but because it has no role, they may not know it's a button.
  • ARIA "Polyfill": Requires role="button", tabindex="0", and custom JS to handle keyboard events. This is reachable via the Tab key and announces correctly, but is more work to build and maintain.
  • Native Button: 0 extra lines of code. Handles focus, Space, Enter, and the "button" role automatically. This is always the preferred approach.

Compliance Success

  • SC 4.1.2: Name, Role, Value - Using a native <button> or role="button" ensures the correct role is announced.
  • SC 2.1.1: Keyboard - Native buttons and properly polyfilled controls are focusable and respond to keyboard input.
