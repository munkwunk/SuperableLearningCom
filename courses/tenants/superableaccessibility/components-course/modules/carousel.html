Carousel / Slider

Carousels are common for showcasing multiple items in a limited space. However, they are frequently inaccessible due to hidden content, lack of keyboard control, or unannounced slide changes.

Practice Task

1. Tab through the carousel controls and activate the "Next" button.

2. Identify the functional cues (Role, State, Name) announced by the screen reader as focus moves into the new slide.

3. Continue tabbing and notice that you can't "accidentally" tab into hidden slides.

Inaccessible Example

In this broken version, all slides are visually hidden but still "visible" to screen readers. There are no controls for keyboard users, and changes aren't announced. It auto-rotates infinitely without a way to stop it.

Slide 1: Universal Design

Why it's broken

  • No Keyboard Controls: Without "Next" or "Previous" buttons, a keyboard user cannot control the carousel.
  • No Pause/Stop: The auto-rotation cannot be stopped, which is a major barrier for users who need more time to read or those with cognitive disabilities.
  • Hidden Content: Off-screen slides should be hidden or inert. In many bad carousels, NVDA still "sees" all items in a long list, confusing the user.
  • Unexpected Changes: Auto-rotation can be extremely disorienting and even dangerous for people with cognitive disabilities or those using screen magnifiers.

Compliance Failures

  • SC 4.1.2: Name, Role, Value - The carousel and its slides lack proper ARIA roles and state information.
  • SC 2.1.1: Keyboard - The carousel cannot be operated via keyboard.
  • SC 2.2.2: Pause, Stop, Hide - Moving content that starts automatically and lasts more than five seconds must have a way to be paused or stopped.

Accessible Example

The accessible version provides explicit controls, uses aria-roledescription to identify the widget, and manages the visibility of slides. Controls are placed at the bottom to preserve natural reading flow.

How it's fixed

  • Aria-roledescription: We use aria-roledescription="carousel" and "slide" to give NVDA users context that this is a specialized widget.
  • Visibility Management: Only the current slide is visible. Others are set to hidden, removing them from both the visual screen and the accessibility tree.
  • Focus Movement: When the user clicks "Next", we move focus to the new slide. This ensures the screen reader immediately begins reading the new content.
  • Logical Control Flow: Controls are placed after the content, allowing learners to encounter the information before deciding to move to the next item.

Compliance Success

  • SC 4.1.2: Name, Role, Value - ARIA roles and roledescriptions provide clear context and state.
  • SC 2.1.1: Keyboard - All carousel functionality is reachable and operable via keyboard.
  • SC 2.2.2: Pause, Stop, Hide - The carousel does not auto-rotate, giving the user full control over the movement of content.
