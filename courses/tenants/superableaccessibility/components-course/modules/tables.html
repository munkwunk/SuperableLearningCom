Interactive Tables

Data tables are essential for presenting structured information. However, when you add interactivity like sorting, it becomes a challenge to keep screen reader users informed of the state changes.

Practice Task

1. Navigate into the table and identify the functional cues that describe the table's dimensions.

2. Activate a sortable header and identify the functional cues (e.g., live region or state attribute) that confirm the new sort order.

3. Use table navigation commands (e.g., Ctrl+Alt+Arrows) and identify how the screen reader announces headers as you move through cells.

Why it's broken

  • No Table Semantics: NVDA sees this as a list of text blocks. It doesn't know it's a table, so it can't navigate by row or column.
  • Missing Header Relationship: There is no scope="col", so the screen reader won't read the header name when you move through the cells.
  • No Sort State: A sighted user sees the items move, but a screen reader user doesn't know which column is sorted or in what direction.

Compliance Failures

  • SC 1.3.1: Info and Relationships - Lack of semantic table markup prevents programmatic determination of headers and data cells.
  • SC 4.1.3: Status Messages - Visual sorting changes are not programmatically announced to the user.

The accessible version uses a real <table> and uses aria-sort on the header buttons.

How it's fixed

  • Semantic Markup: We use <table>, <th>, and <td>. NVDA can now use its table navigation commands (Ctrl+Alt+Arrows).
  • Aria-Sort: The header buttons update their parent's aria-sort attribute to "ascending" or "descending".
  • Polite Announcements: When sorted, a live region confirms the action: "Table sorted by Score, descending order."

Compliance Success

  • SC 1.3.1: Info and Relationships - Using native table elements provides the necessary relationships between headers and cells.
  • SC 4.1.3: Status Messages - Live regions and aria-sort updates provide status feedback when the table state changes.
